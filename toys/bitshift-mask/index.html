The pipeline encodes monthly exposure as a 13-bit mask stored in a smallint:

bit  0     = carryover (active at year start)
bit  1..12 = active during Jan, Feb, ..., Dec

The two formulas in play are:

spell_mask = (1 << (LEAST(end_m, 12) + 1)) - (1 << start_m)
death_mask = mask & ((1 << (K + 1)) - 1)

The rest of the page unpacks both, with interactive bit-row widgets. Throughout, bit 0 is rendered on the right, bit 12 on the left — the natural reading order for binary literals.

1. Binary, smallints, and bit positions

A smallint is a 16-bit two's complement integer. We only use the low 13 bits. Each bit position i contributes 2i to the integer value when it is 1:

value = ∑i biti · 2i

So bit 0 is the ones place, bit 1 the twos, bit 5 the 32s, bit 12 the 4096s. Setting bits 5..8 means the integer equals 25+26+27+28 = 32+64+128+256 = 480.

2. The << operator

x << n shifts the bits of x left by n positions, filling with zeros on the right. Arithmetically it is multiplication by 2n:

x << n  =  x · 2n

The idiom 1 << n therefore yields a one-hot mask with exactly bit n set:

1 << 0  = 1       (bit 0)
1 << 1  = 2       (bit 1)
1 << 5  = 32      (bit 5)
1 << 13 = 8192    (bit 13 — outside our 13-bit window)

3. The range-mask trick: (1 << n) - 1

This is the workhorse. Subtracting one from a power of two flips all bits below it:

1 << 5      =  0b00000100000   (just bit 5)
(1 << 5) - 1 = 0b00000011111   (bits 0..4)

So (1 << n) - 1 produces a mask with bits 0..n-1 set — equivalently, "the low n bits". This is the building block for both formulas in the pipeline.

4. Building a bit range [a..b] (inclusive): the spell formula

Given a spell covering months start_m through end_m (both inclusive, both indices into our 13-bit layout), we want a mask with bits a..b set. The identity is:

bits a..b  =  (bits 0..b)  -  (bits 0..a-1)
           =  ((1 << (b+1)) - 1)  -  ((1 << a) - 1)
           =  (1 << (b+1))  -  (1 << a)

The two -1 terms cancel, leaving the clean form used in SQL:

spell_mask = (1 << (LEAST(end_m, 12) + 1)) - (1 << start_m)

Why it works arithmetically: (1 << (b+1)) is "the bit just above b". Subtracting (1 << a) from it borrows down through every position from a to b, turning all of them into 1s — the same way 1000 - 1 = 999 in decimal cascades a borrow through three digits.

Worked example: start_m=5, end_m=8:

(1 << 9) - (1 << 5)
= 512 - 32
= 480
= 0b0000111100000   (bits 5, 6, 7, 8)

Interactive: the spell formula

If start_m > end_m the formula returns a negative integer — the SQL pipeline guarantees start_m ≤ end_m upstream when spells are constructed.

5. The death cascade: AND with a keep mask

When a death event lands in month K, the person's exposure cannot extend past K. We need to zero out every bit strictly greater than K, keeping bits 0..K untouched.

The keep mask is exactly the range-mask trick again:

keep = (1 << (K + 1)) - 1     -- bits 0..K set

AND with this mask:

death_mask = mask & ((1 << (K + 1)) - 1)

Recall the truth table for AND: a bit survives only if it is 1 in both operands. So bits 0..K survive (where keep is 1), and bits K+1..12 are forced to 0 (where keep is 0). The original bit pattern in positions 0..K is preserved exactly.

Worked example: mask = 0b1111111111111 (all 13 bits), K = 5:

keep = (1 << 6) - 1 = 63 = 0b0000000111111
mask & keep         = 63 = 0b0000000111111

6. Why not NOR? (and AND-NOT equivalence)

NOR is a specific gate: NOR(a, b) = ~(a | b). It returns 1 only where both inputs are 0. That's not what we want here. The pattern "keep some bits, clear others" is AND-NOT at the gate level, sometimes written BIC (bit clear) or ANDN:

mask & ~void   -- keep bits not in void

Where void would be the bits-to-clear (here: bits K+1..12). That is mathematically equivalent to ANDing with the complement, but it costs one extra negation. Going directly to a keep mask is cleaner:

ApproachExpressionOps
AND with keepmask & ((1 << (K+1)) - 1)1 shift, 1 sub, 1 and
AND-NOT voidmask & ~(~0 << (K+1))1 shift, 1 not, 1 and (plus width considerations)
NOR (wrong)~(mask | void)different function — collapses 'keep' bits too
In SQL with a signed smallint the AND-NOT form has to be careful about the width and sign of ~0. The keep-mask form sidesteps that entirely — it never produces a number outside the 13-bit range.

7. BIT_OR aggregate over spells

Per row we have a spell_mask. Per (cpf, cobertura, emp, cod_emp, produto, ref_info) group we want a single mask whose bit i is set iff any spell covered month i. That's the OR aggregate:

SELECT cpf, ..., BIT_OR(spell_mask) AS exposure_mask
FROM spells
GROUP BY cpf, ...

OR's truth table: a bit is 1 in the output iff it is 1 in any input. Aggregating it row-wise is the natural lift to a set of rows. If spells don't overlap, BIT_OR behaves like a disjoint union. If they do overlap, the overlap collapses idempotently — exactly what we want for "was this person exposed in month i?".
